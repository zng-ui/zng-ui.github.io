79/// Widget and property builder types.
80///
81/// # Examples
82///
83/// The example declares a new widget type, `ShowProperties!`, that inherits from `Text!` and display what properties
84/// are set on itself by accessing the [`WidgetBuilder`] at two points. First call is directly in the `widget_intrinsic` that
85/// is called after inherited intrinsics, but before the instance properties are set. Second call is in a build action that is called when
86/// the widget starts building, after the instance properties are set.
87///
88/// [`WidgetBuilder`]: builder::WidgetBuilder
89///
90/// ```
91/// mod widgets {
92///     use std::fmt::Write as _;
93///     use zng::prelude_wgt::*;
94///
95///     #[widget($crate::widgets::ShowProperties)]
96///     pub struct ShowProperties(zng::text::Text);
97///
98///     impl ShowProperties {
99///         fn widget_intrinsic(&mut self) {
100///             let txt = var(Txt::from(""));
101///             widget_set! {
102///                 self;
103///                 txt = txt.clone();
104///             }
105///
106///             let builder = self.widget_builder();
107///
108///             let mut t = Txt::from("Properties set by default:\n");
109///             for p in builder.properties() {
110///                 writeln!(&mut t, "• {}", p.args.property().name).unwrap();
111///             }
112///
113///             builder.push_build_action(move |builder| {
114///                 writeln!(&mut t, "\nAll properties set:").unwrap();
115///                 for p in builder.properties() {
116///                     writeln!(&mut t, "• {}", p.args.property().name).unwrap();
117///                 }
118///                 txt.set(t.clone());            
119///             });
120///         }
121///     }
122/// }
123///
124/// # fn main() {
125/// # let _scope = zng::APP.defaults();
126/// # let _ =
127/// widgets::ShowProperties! {
128///     font_size = 20;
129/// }
130/// # ;
131/// # }
132/// ```
133///
134/// # Full API
135///
136/// See [`zng_app::widget::builder`] for the full API.

139        AnyWhenArcWidgetHandlerBuilder, ArcWidgetHandler, BuilderProperty, BuilderPropertyMut, BuilderPropertyRef, Importance, InputKind,
140        NestGroup, NestPosition, PropertyArgs, PropertyBuildAction, PropertyBuildActionArgs, PropertyBuildActions,
141        PropertyBuildActionsWhenData, PropertyId, PropertyInfo, PropertyInput, PropertyInputTypes, PropertyNewArgs, SourceLocation,
142        WhenBuildAction, WhenInfo, WhenInput, WhenInputMember, WhenInputVar, WidgetBuilder, WidgetBuilderProperties, WidgetBuilding,

246/// Expands a struct to a widget struct and macro.
247///
248/// Each widget is a struct and macro pair of the same name that builds a custom widget using [`WidgetBuilder`]. Widgets
249/// *inherit* from one other widget and can also inherit multiple mix-ins. Widgets can have intrinsic nodes, default properties
250/// and can build to a custom output type.
251///
252/// Properties can be strongly associated with the widget using the `#[property(.., widget_impl(Widget))]` directive, existing properties
253/// can be implemented for the widget using the [`widget_impl!`] macro.
254///
255/// # Attribute
256///
257/// The widget attribute must be placed in a `struct Name(Parent);` declaration, only struct following the exact pattern are allowed,
258/// different struct syntaxes will generate a compile error.
259///
260/// The attribute requires one argument, it must be a macro style `$crate` path to the widget struct, this is used in the generated macro
261/// to find the struct during instantiation. The path must be to the *public* path to the struct, that is, the same path that will be used
262/// to import the widget. After the required widget path [custom rules](#custom-rules) for the generated macro can be declared.
263///
264/// ```
265/// # fn main() { }
266/// use zng::prelude_wgt::*;
267///
268/// /// Minimal widget.
269/// #[widget($crate::Foo)]
270/// pub struct Foo(WidgetBase);
271/// ```
272///
273/// # Inherit
274///
275/// The widget struct field must be the parent widget type. All widgets inherit from another or the
276/// [`WidgetBase`], the parent widgets intrinsic properties and nodes are all included in the new widget. The intrinsic
277/// properties are included by deref, the new widget will dereference to the parent widget, during widget build auto-deref will select
278/// the property methods first, this mechanism even allows for property overrides.
279///
280/// # Intrinsic
281///
282/// The widget struct can define a method `widget_intrinsic` that includes custom build actions in the [`WidgetBuilder`], this special
283/// method will be called once for the widget. The same method is also called for the inherited widgets.
284///
285/// ```
286/// # fn main() { }
287/// use zng::prelude_wgt::*;
288///
289/// #[widget($crate::Foo)]
290/// pub struct Foo(WidgetBase);
291///
292/// impl Foo {
293///     fn widget_intrinsic(&mut self) {
294///         self.widget_builder().push_build_action(|b| {
295///             // push_intrinsic, capture_var.
296///         });
297///     }
298/// }
299/// ```
300///
301/// The example above demonstrates the intrinsic method used to [`push_build_action`]. This is the primary mechanism for widgets to define their
302/// own behavior that does not depend on properties. Note that the widget inherits from [`WidgetBase`], during instantiation
303/// of `Foo!` the base `widget_intrinsic` is called first, then the `Foo!` `widget_intrinsic` is called.
304///
305/// The method does not need to be `pub`, and it is not required.
306///
307/// # Build
308///
309/// The widget struct can define a method that builds the final widget instance.
310///
311/// ```
312/// # fn main() { }
313/// use zng::prelude_wgt::*;
314///
315/// #[widget($crate::Foo)]
316/// pub struct Foo(WidgetBase);
317///
318/// impl Foo {
319///     /// Custom build.
320///     pub fn widget_build(&mut self) -> impl UiNode + use<> {
321///         println!("on build!");
322///         WidgetBase::widget_build(self)
323///     }
324/// }
325/// ```
326///
327/// The build method must have the same visibility as the widget, and can define its own
328/// return type, this is the widget instance type. If the build method is not defined the inherited parent build method is used.
329///
330/// Unlike the [intrinsic](#intrinsic) method, the widget only has one `widget_build`, if defined it overrides the parent
331/// `widget_build`. Most widgets don't define their own build, leaving it to be inherited from [`WidgetBase`]. The base instance type
332/// is an opaque `impl UiNode`.
333///
334/// Normal widgets must implement [`UiNode`], otherwise they cannot be used as child of other widgets.
335/// The widget outer-node also must implement the widget context, to ensure that the widget is correctly placed in the UI tree.
336/// Note that you can still use the parent type build implementation, so even if you need
337/// to run code on build or define a custom type you don't need to deref to the parent type to build.
338///
339/// # Defaults
340///
341/// The [`widget_set!`] macro can be used inside `widget_intrinsic` to set properties and when conditions that are applied on the widget by default,
342/// if not overridden by derived widgets or the widget instance. During the call to `widget_intrinsic` the `self.importance()` value is
343/// [`Importance::WIDGET`], after it is changed to [`Importance::INSTANCE`], so just by setting properties in `widget_intrinsic` they
344/// will have less importance allowing for the override mechanism to replace them.
345///
346/// # Impl Properties
347///
348/// The [`widget_impl!`] macro can be used inside a `impl WgtIdent { }` block to strongly associate a property with the widget,
349/// and the [`property`] attribute has a `widget_impl(WgtIdent)` directive that also strongly associates a property with the widget.
350///
351/// These two mechanisms can be used to define properties for the widget, the impl properties don't need to be imported and are
352/// always selected over other properties of the same name. They also appear in the widget documentation and can have a distinct
353/// visual in IDEs as they are represented by immutable methods while standalone properties are represented by mutable trait methods.
354///
355/// As a general rule only properties that are captured by the widget, or only work with the widget, or have an special meaning in the widget
356/// are implemented like this, standalone properties that can be used in any widget are not implemented.
357///
358/// # Generated Macro
359///
360/// The generated widget macro has the same syntax as [`widget_set!`], except that is also starts the widget and builds it at the end.
361///
362/// This widget macro call:
363///
364/// ```
365/// # use zng::prelude_wgt::*;
366/// # #[widget($crate::Foo)]
367/// # pub struct Foo(WidgetBase);
368/// #
369/// # fn main() {
370/// let wgt = Foo! {
371///     id = "foo";
372/// };
373/// # }
374/// ```
375///
376/// Expands to this:
377///
378/// ```
379/// # use zng::prelude_wgt::*;
380/// # #[widget($crate::Foo)]
381/// # pub struct Foo(WidgetBase);
382/// #
383/// # fn main() {
384/// let wgt = {
385///     let mut wgt = Foo::widget_new();
386///     widget_set! {
387///         &mut wgt;
388///         id = "foo";
389///     }
390///     wgt.widget_build()
391/// };
392/// # }
393/// ```
394///
395/// #### Custom Rules
396///
397/// You can declare custom rules for the widget macro, this can be used to declare custom shorthand syntax for the widget.
398///
399/// The custom rules are declared inside braces after the widget path in the widget attribute. The syntax is similar to `macro_rules!`
400/// rules, but the expanded tokens are the direct input of the normal widget expansion.
401///
402/// ```txt
403/// (<rule>) => { <init> };
404/// ```
405///
406/// The `<rule>` is any macro pattern rule, the `<init>` is the normal widget init code that the rule expands to.
407///
408/// Note that custom rules are not inherited, they apply only to the declaring widget macro, inherited widgets must replicate
409/// the rules if desired.
410///
411/// Example of a widget that declares a shorthand syntax to implicitly set the `id` property:
412///
413/// ```
414/// use zng::prelude_wgt::*;
415///
416/// #[widget($crate::Foo {
417///     ($id:expr) => {
418///         id = $id;
419///     };
420/// })]
421/// pub struct Foo(WidgetBase);
422///
423/// # fn main() {
424/// let wgt = Foo!("foo");
425/// # }
426/// ```
427///
428/// The macro instance above is equivalent to:
429///
430/// ```
431/// # use zng::prelude_wgt::*;
432/// # #[widget($crate::Foo)]
433/// # pub struct Foo(WidgetBase);
434/// #
435/// # fn main() {
436/// let wgt = Foo! {
437///     id = "foo";
438/// };
439/// # }
440/// ```
441///
442/// #### Limitations
443///
444/// The expanded tokens can only be a recursive input for the same widget macro, you can't expand to a different widget.
445///
446/// Some rules are intercepted by the default widget rules:
447///
448/// * `$(#[$attr:meta])* $($property:ident)::+ = $($rest:tt)*`, blocks all custom `$ident = $tt*` patterns.
449/// * `$(#[$attr:meta])* when $($rest:tt)*`, blocks all custom `when $tt*` patterns.
450///
451/// Note that the default single property shorthand syntax is not blocked, for example `Text!(font_size)` will match
452/// the custom shorthand rule and try to set the `txt` with the `font_size` variable, without the shorthand it would create a widget without
453/// `txt` that sets `font_size`. So a custom rule `$p:expr` is only recommended for widgets that have a property of central importance.
454///
455/// # Widget Type
456///
457/// A public associated function `widget_type` is also generated for the widget, it returns a [`WidgetType`] instance that describes the
458/// widget type. Note that this is not the widget instance type, only the struct and macro type. If compiled with the `"inspector"` feature
459/// the type is also available in the widget info.
460///
461/// # See Also
462///
463/// See the [`WidgetBase`], [`WidgetBuilder`], [`WidgetBuilding`], [`NestGroup`] and [`Importance`] for more details.
464///
465/// [`WidgetBuilder`]: builder::WidgetBuilder
466/// [`WidgetType`]: builder::WidgetType
467/// [`WidgetBuilding`]: builder::WidgetBuilding
468/// [`NestGroup`]: builder::NestGroup
469/// [`Importance`]: builder::Importance
470/// [`push_build_action`]: builder::WidgetBuilder::push_build_action
471/// [`UiNode`]: node::UiNode
472/// [`WidgetBase`]: struct@WidgetBase
473/// [`Importance::WIDGET`]: builder::Importance::WIDGET
474/// [`Importance::INSTANCE`]: builder::Importance::INSTANCE
475///
476/// <script>
477/// // hide re-exported docs
478/// let me = document.currentScript;
479/// document.addEventListener("DOMContentLoaded", function() {
480///     while(me.nextElementSibling !== null) {
481///         me.nextElementSibling.remove();
482///     }
483/// });
484/// </script>
485pub use zng_app::widget::widget;

487/// Expands a struct to a widget mix-in.
488///
489/// Widget mix-ins can be inserted on a widgets inheritance chain, but they cannot be instantiated directly. Unlike
490/// the full widgets it defines its parent as a generic type, that must be filled with a real widget when used.
491///
492/// By convention mix-ins have the suffix `Mix` and the generic parent is named `P`. The `P` must not have any generic bounds
493/// in the declaration, the expansion will bound it to [`WidgetImpl`].
494///
495/// # Examples
496///
497/// ```
498/// # fn main() { }
499/// use zng::prelude_wgt::*;
500///
501/// /// Make a widget capable of receiving keyboard focus.
502/// #[widget_mixin]
503/// pub struct FocusableMix<P>(P);
504/// impl<P: WidgetImpl> FocusableMix<P> {
505///     fn widget_intrinsic(&mut self) {
506///         widget_set! {
507///             self;
508///             focusable = true;
509///         }
510///     }
511///     
512///     widget_impl! {
513///         /// If the widget can receive focus, enabled by default.
514///         pub zng::focus::focusable(enabled: impl IntoVar<bool>);
515///     }
516/// }
517///
518/// /// Foo is focusable.
519/// #[widget($crate::Foo)]
520/// pub struct Foo(FocusableMix<WidgetBase>);
521/// ```
522///
523/// The example above declares a mix-in `FocusableMix<P>` and a widget `Foo`, the mix-in is used as a parent of the widget, only
524/// the `Foo! { }` widget can be instantiated, and it will have the strongly associated property `focusable` from the mix-in.
525///
526/// All widget `impl` items can be declared in a mix-in, including the `fn widget_build(&mut self) -> T`. Multiple mix-ins can be inherited
527/// by nesting the types in a full widget `Foo(AMix<BMix<Base>>)`. Mix-ins cannot inherit from other mix-ins.
528///
529/// <script>
530/// // hide re-exported docs
531/// let me = document.currentScript;
532/// document.addEventListener("DOMContentLoaded", function() {
533///     while(me.nextElementSibling !== null) {
534///         me.nextElementSibling.remove();
535///     }
536/// });
537/// </script>
538pub use zng_app::widget::widget_mixin;

540/// Expands a property assign to include an easing animation.
541///
542/// The attribute generates a [property build action] that applies [`Var::easing`] to the final variable inputs of the property.
543///
544/// # Arguments
545///
546/// The attribute takes one required argument and one optional that matches the [`Var::easing`]
547/// parameters. The required first arg is the duration, the second arg is an easing function, if not present the [`easing::linear`] is used.
548///
549/// Some items are auto-imported in each argument scope, [`TimeUnits`] for the first arg and the [`easing`] functions
550/// for the second. This enables syntax like `#[easing(300.ms(), expo)]`.
551///
552/// ## Unset
553///
554/// An alternative argument `unset` can be used instead to remove animations set by the inherited context or styles.
555///
556/// [`TimeUnits`]: zng::layout::TimeUnits
557/// [`easing`]: mod@zng::var::animation::easing
558/// [`easing::linear`]: zng::var::animation::easing::linear
559/// [property build action]: crate::widget::builder::WidgetBuilder::push_property_build_action
560/// [`Var::easing`]: crate::var::Var::easing
561///
562/// ## When
563///
564/// The attribute can also be set in `when` assigns, in this case the easing will be applied when the condition is active, so
565/// only the transition to the `true` value is animated using the conditional easing.
566///
567/// Note that you can't `unset` easing in when conditions, but you can set it to `0.ms()`, if all easing set for a property are `0`
568/// no easing variable is generated, in contexts that actually have animation the `when`` value will be set immediately,
569/// by a zero sized animation.
570///
571/// # Examples
572///
573/// The example demonstrates setting and removing easing animations.
574///
575/// ```
576/// # use zng::prelude_wgt::*;
577/// # #[widget($crate::Foo)] pub struct Foo(WidgetBase);
578/// # #[property(FILL, default(colors::BLACK))]
579/// # pub fn background_color(child: impl UiNode, color: impl IntoVar<Rgba>) -> impl UiNode {
580/// #    child
581/// # }
582/// # #[property(LAYOUT, default(0))]
583/// # pub fn margin(child: impl UiNode, color: impl IntoVar<SideOffsets>) -> impl UiNode {
584/// #    child
585/// # }
586/// # fn main() {
587/// Foo! {

615/// Expands a function to a widget property.
616///
617/// Property functions take one [`UiNode`] child input and one or more other inputs and produces an [`UiNode`] that implements
618/// the property feature.
619///
620/// The attribute expansion does not modify the function, it can still be used as a function directly. Some
621/// properties are implemented by calling other property functions to generate a derived effect.
622///
623/// The attribute expansion generates a hidden trait of the same name and visibility, the trait is implemented for widget builders,
624/// the widget macros use this to set the property. Because it has the same name it is imported together with the property
625/// function, in practice this only matters in doc links where you must use the `fn@` disambiguator.
626///
627/// # Attribute
628///
629/// The property attribute has one required argument and three optional.
630///
631/// #### Nest Group
632///
633/// The first argument is the property [`NestGroup`]. The group defines the overall nest position
634/// of the property, for example, `LAYOUT` properties always wrap `FILL` properties. This is important as widgets are open and any combination
635/// of properties may end-up instantiated in the same widget.
636///
637/// ```
638/// # fn main() { }
639/// use zng::prelude_wgt::*;
640///
641/// #[property(LAYOUT)]
642/// pub fn align(child: impl UiNode, align: impl IntoVar<Align>) -> impl UiNode {
643///     // ..
644/// #   child
645/// }
646/// ```
647///
648/// The nest group can be tweaked, by adding or subtracting integers, in the example bellow both properties are in the `SIZE` group,
649/// but `size` is always inside `max_size`.
650///
651/// ```
652/// # fn main() { }
653/// use zng::prelude_wgt::*;
654///
655/// #[property(SIZE+1)]
656/// pub fn size(child: impl UiNode, size: impl IntoVar<Size>) -> impl UiNode {
657///     // ..
658/// #   child
659/// }
660///
661/// #[property(SIZE)]
662/// pub fn max_size(child: impl UiNode, size: impl IntoVar<Size>) -> impl UiNode {
663///     // ..
664/// #   child
665/// }
666/// ```
667///
668/// #### Default
669///
670/// The next argument is an optional `default(args..)`. It defines the value to use when the property must be instantiated and no value was provided.
671/// The defaults should cause the property to behave as if it is not set, as the default value will be used in widgets that only set the
672/// property in `when` blocks.
673///
674/// ```
675/// # fn main() { }
676/// use zng::prelude_wgt::*;
677///
678/// #[property(FILL, default(rgba(0, 0, 0, 0)))]
679/// pub fn background_color(child: impl UiNode, color: impl IntoVar<Rgba>) -> impl UiNode {
680///     // ..
681/// #   child
682/// }
683/// ```
684///
685/// In the example above the `background_color` defines a transparent color as the default, so if the background color is only set in a `when`
686/// block if will only be visible when it is active.
687///
688/// For properties with multiple inputs the default args may be defined in a comma separated list of params, `default(dft0, ..)`.
689///
690/// #### Impl For
691///
692/// The last argument is an optional `impl(<widget-type>)`, it strongly associates
693/// the property with a widget, users can set this property on the widget without needing to import the property.
694///
695/// Note that this makes the property have priority over all others of the same name, only a derived widget can override
696/// with another strongly associated property.
697///
698/// Note that you can also use the [`widget_impl!`] in widget declarations to implement existing properties for a widget.
699///
700/// #### Capture
701///
702/// After the nest group and before default the `, capture, ` value indicates that the property is capture-only. This flag
703/// changes how the property must be declared, the first argument is a property input and the function can have only one input,
704/// no return type is allowed and the function body must be empty, unused input warnings are suppressed by the expanded code.
705///
706/// Capture-only properties must be captured by a widget and implemented as part of the widget's intrinsics, the reason for
707/// a property function is purely to define the property signature and metadata, the capture-only property function can also
708/// be used to set a property dynamically, such as in a style widget that is applied on the actual widget that captures the property.
709///
710/// A documentation sections explaining capture-only properties is generated for the property, it is also tagged differently in the functions list.
711///
712/// ```
713/// # fn main() { }
714/// use zng::prelude_wgt::*;
715///
716/// /// Children property, must be captured by panel widgets.
717/// #[property(CONTEXT, capture)]
718/// pub fn children(children: impl UiNodeList) { }
719/// ```
720///
721/// # Args
722///
723/// The property function requires at least two args, the first is the child node and the other(s) the input values. The
724/// number and type of inputs is validated at compile time, the types are limited and are identified and validated by their
725/// token name, so you cannot use renamed types.
726///
727/// #### Child
728///
729/// The first function arg must be of type `impl UiNode`, it represents the child node and the property node must
730/// delegate to it so that the UI tree functions correctly. The type must be an `impl` generic, a full path to [`UiNode`]
731/// is allowed, but no import renames as the proc-macro attribute can only use tokens to identify the type.
732///
733/// #### Inputs
734///
735/// The second arg and optional other args define the property inputs. When a property is assigned in a widget only these inputs
736/// are defined by the user, the child arg is provided by the widget builder. Property inputs are limited, and must be identifiable
737/// by their token name alone. The types are validated at compile time, they must be declared using `impl` generics,
738/// a full path to the generic traits is allowed, but no import renames.
739///
740/// #### Input Types
741///
742/// These are the allowed input types:
743///
744/// ##### `impl IntoVar<T>`
745///
746/// The most common type, accepts any value that can be converted [`IntoVar<T>`], usually the property defines the `T`, but it can be generic.
747/// The property node must respond to var updates. The input kind is [`InputKind::Var`]. No auto-default is generated for this type, property
748/// implementation should provide a default value that causes the property to behave as if it was not set.
749///
750/// The input can be read in `when` expressions and can be assigned in `when` blocks.
751///
752/// ##### `impl IntoValue<T>`
753///
754/// Accepts any value that can be converted [`IntoValue<T>`] that does not change, usually the property
755/// defines the `T`, but it can be generic. The input kind is [`InputKind::Value`]. No auto-default is generated for this type.
756///
757/// The input can be read in `when` expressions, but cannot be assigned in `when` blocks.
758///
759/// ##### `impl UiNode`
760///
761/// This input accepts another [`UiNode`], the implementation must handle it like it handles the child node, delegating all methods. The
762/// input kind is [`InputKind::UiNode`]. The [`NilUiNode`] is used as the default value if no other is provided.
763///
764/// The input cannot be read in `when` expressions, but can be assigned in `when` blocks.
765///
766/// ##### `impl UiNodeList`
767///
768/// This input accepts an [`UiNodeList`], the implementation must handle it like it handles the child node, delegating all methods. The
769/// input kind is [`InputKind::UiNodeList`]. An empty list is used as the default value if no other is provided.
770///
771/// The input cannot be read in `when` expressions, but can be assigned in `when` blocks.
772///
773/// ##### `impl WidgetHandler<A>`
774///
775/// This input accepts any [`WidgetHandler<A>`] for the argument type `A`, usually the property defines the `A`, but it can be generic.
776/// The input kind is [`InputKind::WidgetHandler`]. A no-op handler is used for the default if no other is provided.
777///
778/// Event handler properties usually have the `on_` name prefix. You can use the [`event_property!`] macro to generate standard event properties.
779///
780/// The input cannot be read in `when` expressions, but can be assigned in `when` blocks.
781///
782/// # Getter Properties
783///
784/// Most properties with var inputs are *setters*, that is the inputs affect the widget. Some properties
785/// can be *getters*, detecting widget state and setting it on the *input* variable. These properties are usually named with
786/// a prefix that indicates their input is actually for getting state, the prefixes `is_` and `has_` mark a property with
787/// a single `bool` input that reads a widget state, the prefix `get_` and `actual_` marks a property that reads a non-boolean state from
788/// the widget.
789///
790/// Getter properties are configured with a default read-write variable, so that they can be used in `when` expressions directly,
791/// for example, `when *#is_pressed`, the `is_pressed` property has a `default(var(false))`, so it automatically initializes
792/// with a read-write variable that is used in the when condition. The property attribute generates defaults automatically
793/// based on the prefix, the default is `var(T::default())`, this can be overwritten just by setting the default,
794/// it is not possible to declare a getter property without default.
795///
796/// Note that if a property is used in `when` condition without being set and without default value the when block is discarded on
797/// widget build. If you are implementing a getter property that is not named using the prefixes listed above you must set `default(var(T::default())`.
798///
799/// # Generics
800///
801/// Apart from the `impl` generics of inputs and child, there is some support for named generic types, only one named generic is allowed
802/// for inputs `impl IntoVar<T>`, `impl IntoValue<T>` and `impl WidgetHandler<A>`.
803///
804/// # Output
805///
806/// The property output type must be any type that implements [`UiNode`], usually an opaque type `impl UiNode` is used. The property
807/// node can be anything, as long as it delegates to the child node, see [`match_node`] or [`ui_node`] about implementing a node.
808///
809/// Some common property patterns have helper functions, for example, to setup a context var you can use the [`with_context_var`] function.
810///
811/// # More Details
812///
813/// See [`property_id!`] and [`property_args!`] for more details about what kind of meta-code is generated for properties.
814///

846/// Expands an impl block into an [`UiNode`] trait implementation.
847///
848/// Missing [`UiNode`] methods are generated by this macro. The generation is configured in the macro arguments.
849/// The arguments can be a single keyword, a delegate or an entire struct declaration.
850///
851/// The general idea is you implement only the methods required by your node and configure this macro to generate the methods
852/// that are just boilerplate UI tree propagation, and in [new node](#new-node) mode var and event handlers can be inited automatically
853/// as well.
854///
855/// # Delegate to single `impl UiNode`
856///
857/// If your node contains a single child node, you can configure the attribute
858/// to delegate the method calls for the child node.
859///
860/// ```
861/// # fn main() { }
862/// use zng::prelude_wgt::*;
863///
864/// struct MyNode<C> {
865///     child: C
866/// }
867/// #[ui_node(delegate = &mut self.child)]
868/// impl<C: UiNode> UiNode for MyNode<C> { }
869/// ```
870///
871/// If the child node is in a field named `child` you can use this shorthand to the same effect:
872///
873/// ```
874/// # fn main() { }
875/// # use zng::prelude_wgt::*;
876/// # struct MyNode<C> { child: C }
877/// #[ui_node(child)]
878/// impl<C: UiNode> UiNode for MyNode<C> { }
879/// ```
880///
881/// The generated code simply calls the same [`UiNode`] method in the child.
882///
883/// # Delegate to a `impl UiNodeList`
884///
885/// If your node contains multiple children nodes in a type that implements [`UiNodeList`],
886/// you can configure the attribute to delegate to the equivalent list methods.
887///
888/// ```
889/// # fn main() { }
890/// use zng::prelude_wgt::*;
891///
892/// struct MyNode<L> {
893///     children: L
894/// }
895/// #[ui_node(delegate_list = &mut self.children)]
896/// impl<L: UiNodeList> UiNode for MyNode<L> { }
897/// ```
898///
899/// If the children list is a member named `children` you can use this shorthand to the same effect:
900///
901/// ```
902/// # fn main() { }
903/// # use zng::prelude_wgt::*;
904/// # struct MyNode<L> { children: L }
905/// #[ui_node(children)]
906/// impl<L: UiNodeList> UiNode for MyNode<L> { }
907/// ```
908///
909/// The generated code simply calls the equivalent [`UiNodeList`] method in the list.
910/// That is the same method name with the `_all` prefix. So `UiNode::init` maps to `UiNodeList::init_all` and so on.
911///
912/// ## Don't Delegate
913///
914/// If your node does not have any child nodes you can configure the attribute to generate empty missing methods.
915///
916/// ```
917/// # fn main() { }
918/// # use zng::prelude_wgt::*;
919/// # struct MyNode { }
920/// #[ui_node(none)]
921/// impl UiNode for MyNode { }
922/// ```
923///
924/// The generated [`measure`] and [`layout`] code returns the fill size.
925///
926/// The other generated methods are empty.
927///
928/// # Validation
929///
930/// If delegation is configured but no delegation occurs in the manually implemented methods
931/// you get the error ``"auto impl delegates call to `{}` but this manual impl does not"``.
932///
933/// To disable this error use `#[allow_(zng::missing_delegate)]` in the method or in the `impl` block. The
934/// error is also not shown if the method body contains a call to the [`todo!()`] macro.
935///
936/// The [`measure`] method is an exception to this and will not show the error, its ideal implementation
937/// is one where the entire sub-tree is skipped from the computation.
938///
939/// # Mixing Methods
940///
941/// You can use the same `impl` block to define [`UiNode`] methods and
942/// associated methods by using this attribute in a `impl` block without trait. The [`UiNode`]
943/// methods must be tagged with the `#[UiNode]` pseudo-attribute.
944///
945/// ```
946/// # fn main() { }
947/// # use zng::prelude_wgt::*;
948/// # struct MyNode { child: BoxedUiNode }
949/// #[ui_node(child)]
950/// impl MyNode {
951///     fn do_the_thing(&mut self) {
952///         // ..
953///     }
954///
955///     #[UiNode]
956///     fn init(&mut self) {
957///         self.child.init();
958///         self.do_the_thing();
959///     }
960///
961///     #[UiNode]
962///     fn update(&mut self, updates: &WidgetUpdates) {
963///         self.child.update(updates);
964///         self.do_the_thing();
965///     }
966/// }
967/// ```
968///
969/// The above code expands to two `impl` blocks, one with the associated method and the other with
970/// the [`UiNode`] implementation.
971///
972/// This is particularly useful for nodes that have a large amount of generic constraints, you just type them once.
973///
974/// # New Node
975///
976/// In all the usage seen so far you must declare the `struct` type and the generic bounds to
977/// make it work in the `impl` block, and any var or event in it needs to be subscribed manually. You can
978/// avoid this extra boilerplate by declaring the node `struct` as an arg for the macro.
979///
980/// ```
981/// # fn main() { }
982/// # use zng::prelude_wgt::*;
983/// fn my_widget_node(child: impl UiNode, number: impl IntoVar<u32>) -> impl UiNode {
984///     #[ui_node(struct MyNode {
985///         child: impl UiNode,
986///         #[var] number: impl Var<u32>,
987///     })]
988///     impl UiNode for MyNode {
989///         fn update(&mut self, updates: &WidgetUpdates) {
990///             self.child.update(updates);
991///             if let Some(n) = self.number.get_new() {
992///                 println!("new number: {n}");
993///             }
994///         }
995///     }
996///     MyNode {
997///         child,
998///         number: number.into_var(),
999///     }
1000/// }
1001/// ```
1002///
1003/// In the example above the `MyNode` struct is declared with two generic params: `T_child` and `T_var`, the unimplemented
1004/// node methods are delegated to `child` because of the name, and the `number` var is subscribed automatically because of
1005/// the `#[var]` pseudo attribute.
1006///
1007/// Note that you can also use [`node::match_node`] to declare *anonymous* nodes, most of the properties are implemented using
1008/// this node instead of the `#[ui_node]` macro.
1009///
1010/// #### Generics
1011///
1012/// You can declare named generics in the `struct`, those are copied to the implement block, you can also have members with type
1013/// `impl Trait`, a named generic is generated for these, the generated name is `T_member`. You can use named generics in the `impl`
1014/// generics the same way as you would in a function.
1015///
1016/// #### Impl Block
1017///
1018/// The impl block cannot have any generics, they are added automatically, the `UiNode for` part is optional, like in the delegating
1019/// mode, if you omit the trait you must annotate each node method with the `#[UiNode]` pseudo attribute.
1020///
1021/// #### Delegation
1022///
1023/// Delegation is limited to members named `child` or `children`, there is no way to declare a custom delegation in *new node*
1024/// mode. If no specially named member is present the `none` delegation is used.
1025///
1026/// #### Subscription
1027///
1028/// You can mark members with the `#[var]` or `#[event]` pseudo attributes to generate initialization code that subscribes the var or
1029/// event to the [`WIDGET`] context. The init code is placed in a method with signature `fn auto_subs(&mut self)`,
1030/// if you manually implement the `init` node method you must call `self.auto_subs();` in it, a compile time error is emitted if the call is missing.
1031///
1032/// #### Limitations
1033///
1034/// The new node type must be private, you cannot set visibility modifiers. The struct cannot have any attribute set on it, but you can
1035/// have attributes in members, the `#[cfg]` attribute is copied to generated generics. The `impl Trait` auto-generics only works for
1036/// the entire type of a generic, you cannot declare a type `Vec<impl Debug>` for example.
